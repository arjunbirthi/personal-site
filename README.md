# Personal Site

Personal site of Arjun Birthi. Jekyll-powered, minimalist literary aesthetic — parchment palette, Lora serif.

## Running locally

Prerequisite: Ruby 3.3.0 via rbenv (already set up in this directory).

```bash
bundle install            # First time only
bundle exec jekyll serve  # Dev server at http://localhost:4000 with auto-reload
```

If `bundle` errors with a Ruby version mismatch, your shell isn't picking up rbenv shims — prepend `PATH="$HOME/.rbenv/shims:$PATH"` to the command, or restart your shell so rbenv's init runs.

## Writing a new article

### 1. Create a Markdown file in `_posts/`

Jekyll **requires** post filenames to follow this exact format:

```
YEAR-MONTH-DAY-title.MARKUP
```

Where `YEAR` is a four-digit number, `MONTH` and `DAY` are both two-digit numbers, and `MARKUP` is the file extension (use `.md` or `.markdown`). Examples:

```
2026-05-12-on-renewable-energy.md
2026-06-01-clavicular-forms.md
```

The date in the filename is what Jekyll uses to sort posts and build URLs — it must match the date in the front matter.

### 2. Add front matter at the top

```yaml
---
layout: post
title: "Your title here"
date: 2026-05-12
tag: essay
---
```

| Field | Required | Notes |
|-------|----------|-------|
| `layout` | yes | Always `post`. Wraps your markdown in `_layouts/post.html`. |
| `title` | yes | Renders as the article's H1 and in the homepage list. |
| `date` | yes | ISO date, optionally with time + offset: `2026-05-12 14:30:00 -0400`. |
| `tag` | no | Single short label (`essay`, `opinion`, `note`). Renders as the pill on the right of the homepage list and in the article header. Omit if you don't want a tag. |

### 3. Write Markdown below the front matter

Headings (`##`, `###`), emphasis (`*italic*`, `**bold**`), links, blockquotes (`> ...`), bulleted/numbered lists, inline `` `code` ``, and fenced code blocks all render with custom styling defined in `style.css`.

For syntax-highlighted code, use standard fenced blocks:

````markdown
```ruby
def hello(name)
  puts "Hi, #{name}"
end
```
````

Or Jekyll's Liquid `highlight` tag:

```liquid
{% highlight ruby %}
def hello(name)
  puts "Hi, #{name}"
end
{% endhighlight %}
```

### 4. Save

If `jekyll serve` is running, the site auto-rebuilds and the post appears at the top of the homepage list (newest first).

## Project layout

```
index.html              Homepage entry. Trivial — just `layout: home`.
_layouts/default.html   Shared head + body chrome. Loads Lora font and style.css.
_layouts/home.html      Masthead + dynamic post list. Inherits from default.
_layouts/post.html      Article header + body. Inherits from default.
_posts/                 Articles. One Markdown file per post.
style.css               All visual styling. Single source of truth for the design.
about.markdown          About page, served at /about/.
404.html                Error page.
_config.yml             Site metadata and Jekyll build settings.
.gitignore              Excludes _site/, vendor/, Jekyll caches.
Gemfile / Gemfile.lock  Ruby gem dependencies (Jekyll 4.4, jekyll-feed).
```

`_site/` is the build output — generated by Jekyll, gitignored, never edited by hand.

### How the layouts compose

`_layouts/default.html` holds the shared chrome (head, fonts, parchment body, container div). `home` and `post` declare `layout: default` in *their* front matter, so their content gets injected into default's `{{ content }}` slot. Pages and posts then declare `layout: home` or `layout: post`, and the chain composes outward.

This means: change the font, background, or any site-wide chrome once in `_layouts/default.html`. Change article structure once in `_layouts/post.html`. Change visual styling once in `style.css`. Each concern has one home.

## Design tokens

Defined in `style.css`. Reach for these when adding new styles instead of inventing new values:

| Token             | Value      | Used for |
|-------------------|------------|----------|
| Background        | `#F5F0E8`  | Parchment body color |
| Body text (ink)   | `#2C2518`  | Default text and link color |
| Muted text        | `#6d6d6d`  | Dates, back-link, metadata |
| Heading muted     | `#4A4A4A`  | Site name, article H2, blockquote |
| Soft border       | `#C8BFB0`  | Tag pills, blockquote rule, body hr |
| Subtle border     | `#D9D0C0`  | Section dividers (page-level hr) |
| Code background   | `#EDE5D5`  | Inline `code` and `<pre>` blocks |
| Top accent bar    | `#7a9fe4`  | 6px blue stripe at top of body |
| Body font         | Lora       | Loaded from Google Fonts |
| Body size         | 16px / 1.7 | line-height |

## Deployment (when ready)

**Plan: GitHub Pages via GitHub Actions.** Free for public repos.

Why Actions and not Pages' built-in Jekyll builder? Because the built-in builder pins Jekyll to 3.10, but this site uses Jekyll 4.4. GitHub Actions runs your build pipeline on GitHub's runner, so version constraints don't apply.

When ready to deploy:
1. Push this repo to GitHub (`gh repo create` or via the web UI).
2. Add a GitHub Actions workflow at `.github/workflows/jekyll.yml` that runs `bundle exec jekyll build` and deploys `_site/` to Pages.
3. In the repo's Settings → Pages → Source, choose "GitHub Actions".

Costs:
- GitHub Pages: free for public repos.
- GitHub Actions: unlimited free minutes on public repos. (Private repos get 2,000 free minutes/month — a Jekyll build is ~30 seconds, so even daily pushes use only ~15 min/month.)
